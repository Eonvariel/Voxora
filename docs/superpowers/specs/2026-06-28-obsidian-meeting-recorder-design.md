# Obsidian 会议录音转写与 AI 分析插件设计

## 1. 目标

第一版构建一个桌面端 Obsidian 插件，服务个人用户在会议中录音、准实时转写，并在会议结束后生成结构化 Markdown 会议笔记。产品方向是“可靠录音笔”：优先保证录音不丢、转写可恢复、会议内容稳定沉淀到当前 Vault。

成功标准：

- 30 分钟中文会议可以稳定跑通。
- 录音文件可保存到用户选择的 Vault 目录。
- 准实时转写最终确认段可以持续写入会议笔记。
- 会议结束后 1-2 分钟内生成可用 Markdown 纪要。
- 网络或转写服务中断时不丢录音，恢复后可补转写并重新分析。

## 2. 用户与范围

目标用户是个人 Obsidian 用户，尤其是需要把会议内容沉淀为个人知识库资产的人。第一版先自用，跑通后再考虑公开发布。

第一版支持：

- 桌面端 Obsidian。
- 麦克风实时录音。
- 音频输入设备选择。
- 准实时分段转写。
- 会议结束后自动 AI 分析。
- 从已有录音文件补转写或重新转写。
- 手动重新分析。

第一版明确不支持：

- 移动端。
- 团队协作。
- 会议历史管理页。
- 说话人识别。
- 系统音频捕获。
- 逐字字幕级流式显示。
- 多模板或模板库。
- 多场并行录音。

## 3. 核心用户流程

1. 用户通过 Ribbon 图标、命令面板命令或侧边栏录音面板开始录音。
2. 开始录音前，用户可输入会议标题；未填写时使用 `会议记录 YYYY-MM-DD HH-mm`。
3. 插件使用用户选择或默认的麦克风输入设备开始录音。
4. 插件立即在用户配置的会议笔记目录中创建 Markdown 笔记。
5. 录音过程中，侧边栏显示临时识别结果；Markdown 只追加最终确认段落。
6. 用户可暂停、继续、停止录音。
7. 用户可点击“标记重点”，插件在当前时间戳记录一个重点标记。
8. 停止录音后，插件自动触发 AI 分析。
9. AI 分析结果写入同一篇会议笔记的 AI 分析区块。
10. 用户后续可重新分析；重新分析只替换 AI 分析区块，不改完整转写。

同一时间只允许一场活跃录音。

## 4. 文件与笔记结构

会议笔记保存到用户选择的 Vault 目录。录音文件默认保存到用户选择的 Vault 附件目录，用户可在设置中关闭保存录音。

每场会议生成一篇主 Markdown 笔记，录音作为附件被链接到该笔记。笔记包含少量 YAML frontmatter：

```yaml
---
title: 会议记录 2026-06-28 15-30
startedAt: 2026-06-28T15:30:00+08:00
endedAt: 2026-06-28T16:00:00+08:00
durationSeconds: 1800
status: completed
audioPath: Attachments/Meetings/meeting-2026-06-28-15-30.webm
transcriptionProvider: alibaba-bailian
analysisProvider: openai-compatible
analysisModel: qwen-or-compatible-model
---
```

正文采用固定区块：

```markdown
# 会议记录 2026-06-28 15-30

<!-- voxora-analysis:start -->
## AI 分析

### 摘要

### 关键决策

### 行动项

### 待跟进问题
<!-- voxora-analysis:end -->

## 会议信息

- 录音：[[meeting-2026-06-28-15-30.webm]]
- 开始时间：2026-06-28 15:30
- 结束时间：2026-06-28 16:00
- 状态：已完成

## 重点标记

- [00:12:34] 重点标记

<!-- voxora-transcript:start -->
## 完整转写

[00:00:05] 这里写入最终确认的转写段落。

[00:00:12] 这里写入下一段最终确认的转写段落。
<!-- voxora-transcript:end -->
```

每个最终确认段带粗粒度时间戳。重新分析只替换 `voxora-analysis` 区块，不修改转写区块。

## 5. 服务商与配置

转写链路第一版只实现阿里百炼语音模型 Provider。插件产品上不额外承诺复杂语言能力；识别语言和模型能力跟随阿里百炼语音模型本身。

AI 分析链路使用 OpenAI-compatible 配置，用户提供：

- `baseUrl`
- `apiKey`
- `model`

转写服务和 AI 分析服务分开配置，允许用户自由组合。

API Key 第一版保存在插件设置中。设置页必须明确提示：如果 Vault 或插件配置参与同步，API Key 可能被同步或泄露。

设置页第一版只包含启动必需配置：

- 录音保存目录。
- 会议笔记目录。
- 默认输入设备。
- 是否保存录音。
- 阿里百炼语音配置。
- OpenAI-compatible 分析配置。
- 自动分析开关。

AI 分析模板第一版固定，不支持用户自定义 Prompt、多模板或模板库。

## 6. 录音、转写与状态

录音控制支持：

- 开始。
- 暂停。
- 继续。
- 停止。

转写体验：

- 侧边栏可显示临时识别结果。
- Markdown 只写入最终确认段落。
- 最终确认段落追加写入，不频繁重写整篇笔记。
- 转写失败区间在笔记中明确标记。

插件状态至少包括：

- idle：未录音。
- starting：准备录音和创建笔记。
- recording：录音中。
- paused：暂停中。
- stopping：停止中。
- analyzing：AI 分析中。
- completed：已完成。
- interrupted：异常中断。
- failed：无法自动恢复的失败状态。

## 7. 可靠性与恢复

可靠性原则：转写可以降级，录音不能丢。

网络断开或转写服务失败时：

- 录音继续保存。
- 本地继续缓冲音频。
- 转写连接自动重试。
- Markdown 标记失败区间。
- 停止后允许补转写或重新转写。
- 补转写完成后允许重新分析。

Obsidian 重启后：

- 不尝试恢复正在录音的进程。
- 保留已经写入的笔记和录音文件。
- 将会议状态标记为异常中断。
- 允许用户从保存的录音补转写并重新分析。

## 8. UI 入口

第一版包含：

- Ribbon 图标：快速打开或启动录音流程。
- 命令面板命令：开始录音、停止录音、重新分析、从录音补转写等。
- 侧边栏录音面板：显示设备、状态、计时、临时识别结果、开始/暂停/继续/停止按钮、标记重点按钮。

插件提供命令，用户可在 Obsidian 中自行绑定快捷键；第一版不设置默认快捷键，不提供全局系统快捷键。

## 9. 架构

建议模块边界：

- `RecordingController`：管理录音状态机，保证同一时间只允许一场活跃录音。
- `AudioCapture`：处理输入设备选择、麦克风采集、本地缓存和录音文件保存。
- `TranscriptionProvider`：定义转写适配器接口；第一版实现阿里百炼语音转写。
- `MeetingNoteWriter`：创建 Markdown、追加最终转写段、插入重点标记、更新 frontmatter、替换 AI 分析区块。
- `AnalysisProvider`：实现 OpenAI-compatible 分析接口，输入完整转写和重点标记，输出固定结构纪要。
- `RecoveryService`：发现异常中断会议、失败区间和残留音频，提供补转写和重新分析入口。
- `RecorderView`：实现侧边栏 UI。
- `SettingsTab`：实现必需配置和 API Key 风险提示。

Provider 接口要隔离外部服务差异，使核心状态机和笔记写入逻辑可以用 mock Provider 测试。

## 10. 测试策略

自动测试重点：

- 录音状态机：开始、暂停、继续、停止、异常中断、重复启动保护。
- Markdown 写入：创建笔记、追加最终段、插入重点标记、更新时间戳。
- 区块替换：重新分析只替换 AI 分析区块，不改完整转写。
- frontmatter 更新：状态、时间、音频路径、服务商和模型字段正确。
- 失败恢复：转写断线、重试、失败区间标记、补转写入口。

外部服务测试：

- 阿里百炼转写使用 mock Provider 覆盖核心流程。
- OpenAI-compatible 分析使用 mock Provider 覆盖成功、失败、超时。
- 真实 API 只做少量手工验收，不作为主要自动测试依赖。

- 用户可以配置阿里百炼语音转写。
- 用户可以配置 OpenAI-compatible AI 分析。
- 开始录音时会创建会议笔记。
- 录音时侧边栏显示临时识别结果。
- 最终确认转写段持续追加到 Markdown。
- 每个转写段包含时间戳。
- 用户可以暂停、继续、停止。
- 用户可以插入重点标记。
- 停止后自动生成 AI 分析。
- 重新分析只替换 AI 分析区块。
- 断网或转写失败时录音不丢。
